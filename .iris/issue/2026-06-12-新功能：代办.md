# 新功能：代办

很简单，就是活动issue中所有还没有打勾的待办项，有一个专门的面板管理，你认为应该怎么实现？

然后最好是还有一些新功能标签，其实也是一个简单的元数据，然后其实就是，最好去查一下LINEAR 支持什么，然后抄一部分过来

## 调研结论（2026-06-12，agent 调研）

### 一、待办面板怎么实现

数据源：活动 issue 正文里的 GFM 任务列表项（`- [ ]` / `- [x]`），按字面解析，符合渲染层确定性红线。

**读侧（关键发现：几乎免费）**。`iris-scanner.ts` 的 `readDocMeta` 本来就把每个文档全文读进内存（为了解析 frontmatter），提取任务项不增加任何 IO。做法：扫描时逐行匹配任务项语法，跳过 fenced code block 内的行（需要追踪 ``` 围栏状态，纯确定性，无启发式），给 `IrisDoc` 加一个字段：

```ts
todos: { line: number; checked: boolean; text: string }[]
```

文件监听回路（chokidar → 中断 → 重扫）已存在，面板自动保持实时，不需要新机制。

**面板**。新增一个集合视图（复用 `collectDocs` + IssuePanel 的骨架）：只收集非归档、`isActiveIssue` 为真的 issue 的未勾选项；列为「待办内容 / 来源 issue / 工作区」，按来源 issue 分组；点行进入单篇视图。

**写侧（面板上直接打勾）**。完全照 `issue-actions.ts` 的 `setDocStatus` 模式，但手术对象从 frontmatter 行换成正文行：

1. 文档已在编辑器打开 → 走 editor session（避免与编辑器竞态）；
2. 否则读盘 → 校验第 N 行仍与扫描时记录的原文一致（不一致 = 文档已变，拒绝盲改，等下次扫描）→ 仅把该行 `[ ]` 改 `[x]` → `doc.save`。

单行字节级手术，零 diff 纪律天然满足。不需要新指令，复用 `doc.save`。

### 二、Linear 元数据体系（抄什么）

Linear issue 的属性全集：labels（支持分组、颜色、workspace/team 两级词表）、priority（Urgent/High/Medium/Low/None）、estimate、due date、assignee、project、cycle、relations（blocks/duplicate/related）、sub-issues、triage。

**建议抄**：

- **labels** — frontmatter 单行 YAML flow 序列：`labels: [bug, ui]`。gray-matter 能解析，单行所以 `setFrontmatterKey` 的行级手术仍适用（需小扩展：允许写入不被引号包裹的 flow 序列字面量）。**词表不建注册表**（红线）：标签集合 = 全项目文档中实际出现的标签的并集，按工作区推断的同款思路；自动补全来自这个并集；颜色用标签名确定性哈希映射到 Rose Pine 强调色板。最后一个使用消失 = 标签消失，无管理 UI。
- **priority** — 可选第二步：`priority: high`，软值有序集（urgent/high/medium/low），照 `SOFT_STATUSES` 下拉的同款做法，issue 面板按它排序。

**不抄**：estimate / cycle（无迭代概念）、project(workspace 已覆盖)、assignee（单人 + agent 场景暂无意义）、relations / sub-issues（未来可由 `[[链接]]` 承担）、triage / SLA。

issue 面板读侧几乎免费：`IrisDoc.frontmatter` 已携带完整解析结果，加一列 + 过滤 chips 即可。

### 三、待用户拍板的产品决策

1. 待办面板的入口：左栏独立集合视图，还是 issue 面板内的一个 tab？
2. 面板上直接勾选（写侧手术），还是 v1 先只读跳转？
3. 提取范围：仅活动 issue，还是所有类型文档？
4. 标签第一步只做 labels，还是 labels + priority 一起？
5. 里程碑安排：M6 是 v1 收口，本功能进 M6 还是排 v1 之后？

## 实现记录（2026-06-12，用户授权 agent 自行裁决）

上述五个决策的裁决：

1. 入口 = 左栏树上方一行固定的「待办」入口（带计数），点击打开中栏待办面板。它是横切的注意力视图，不属于任何单一类型的集合，所以不做成 issue 面板的 tab。
2. 直接勾选。写侧按调研方案：读盘 → 校验该行与扫描时记录的原文逐字节一致（不一致拒绝盲改，等下次扫描）→ 仅改该行 `[ ]` → `[x]` → `doc.save`。目标文档在编辑器打开时先 flush 未保存编辑，写盘后由既有 ISR 把干净会话实时重载。
3. 仅活动（非归档 + 未解决）issue——与左栏透镜口径一致。
4. labels + priority 一起做了（priority 完全复用 status 的下拉模式，增量很小）。
5. 未动 roadmap；本功能作为 dogfood 需求独立完成，里程碑归属留给用户定。

落地内容：

- 读侧：`IrisDoc` 新增 `todos`（GFM 任务项逐行字面提取，跳过 fenced code，记录行号与原文）与 `labels`（YAML 序列 → 项；单标量 → 单元素；无逗号拆分启发式）。提取逻辑在 `shared/markdown-utils`（`extractTodos`），scanner 填充，监听回路自动保持实时。
- 待办面板：`TodoPanel` 按来源 issue 分组，勾选写回源文档，行内点击跳转单篇视图；左栏入口带总数。
- 标签：issue 面板新增标签列（点击 chip 过滤）；TypedHeader 对 issue 文档提供标签编辑（候选 = 全项目标签并集，无注册表）；颜色 = 标签名确定性哈希 → Rose Pine 强调色。frontmatter 写入用单行 flow 序列（`labels: [bug, ui]`），行级手术仍适用。
- 优先级：`priority:` 软值（urgent/high/medium/low 候选，自由值放行），issue 面板列 + 下拉、TypedHeader 字段，活动列表按优先级 → 日期排序，无值排最后。

验证：vitest 43 通过（新增 extractTodos / flow-seq 读写 / scanner 投影用例），三进程 typecheck 干净，build + 启动冒烟通过。**待手工验收后再置 done。**
