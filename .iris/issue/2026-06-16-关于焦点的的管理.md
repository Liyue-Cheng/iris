## 检查结论（窗口/键盘输入焦点）

本次检查的是**窗口内的键盘输入焦点**（光标/打字焦点该落在哪个控件），不是"选中哪篇文档"。

### 根因：没有焦点协调器，焦点被绑死在各组件的"挂载生命周期"上

全仓没有任何统一裁决"此刻键盘焦点该落在三栏的哪里"的代码——没有全局快捷键在栏间移动焦点（`src/renderer/components/layout` 与 `App` 里搜不到 `keydown`/`ctrlKey`），也没有窗口级的焦点恢复。每个组件各自在自己 mount/reveal 的时机里硬调 `.focus()`：

- 终端：`TerminalView.tsx` 里有 8+ 处 `term.focus()`，且每次重挂都无条件抢焦点。
- 编辑器：只有"新建文档"这一条路径会主动聚焦（`focusOnMount` 一次性标志）。
- 集合面板：靠 `tabIndex` 等用户先点一下。

后果就是你说的"该自动切的时候没及时切"——下面逐条列出，均带 `文件:行` 证据与现象。

### 问题清单

**P1 — 点开已存在文档后，光标不会自动落进正文（最常见）**
- 只有新建文档才传 `focusEditor`：`CreateDocDialog.tsx:75` → `editor-store.ts:153 focusOnMount` → `CrepeEditor.tsx:91 view.focus()`。
- 普通选中文档的所有入口都**不**传 `focusEditor`：`project-store.ts:166 selectDoc`、`LensTree.tsx:104`、`RawTree.tsx:41`、`IssuePanel.tsx:256`（按 Enter 打开）、`TodoPanel`、`SimpleList.tsx:71`。
- 现象：点开任何已有文档后必须再用鼠标点一下编辑器才能打字。

**P2 — 切到终端会话后，焦点被整段 scrollback 重放阻塞 100–500ms，期间按键被丢弃**
- 终端容器在重放完成前是 `inert`（`TerminalView.tsx:794-799`），焦点只在 reveal 之后的一帧才发生（`TerminalView.tsx:808-816`）。
- reveal 只在 `finishReplay()` 之后触发（`TerminalView.tsx:623-652`），重放耗时按注释为 100–500ms（`TerminalView.tsx:791`），异常时要等到失败兜底超时 `REPLAY_FAILSAFE_MS` 才 reveal（`TerminalView.tsx:640-647, 756`）。
- 注释明说 inert 期间按键"must not leak"——即被有意吞掉（`TerminalView.tsx:791-792`）。
- 现象：切到一个会话后有可感知延迟才能打字；重放慢/卡时延迟更久，这段时间里敲的键直接丢失。

**P3 — 上一篇文档有未保存编辑时，整条焦点切换被一次写盘 IPC 串行阻塞**
- 切换前先 `await editorStore.flushBeforeSwitch()`（`project-store.ts:174`）→ `save()` → `await pipeline.dispatch('doc.save')` 磁盘往返（`editor-store.ts:248`）之后才 setState 切换。
- `selectRoot` / `openCollection` / `openTodos` / `openSession` 同样先 await flush（`project-store.ts:131,137,144`、`session-actions.ts:16`）。
- （`save()` 在不脏时立即返回 `editor-store.ts:242`，所以只在上一篇有未保存改动时才发生。）
- 现象：在脏文档上点别的文档/根/集合，会先等一次写盘才开始切换，左栏高亮、右栏会话、编辑器挂载、焦点全被延后。

**P4 — 打开 Issue 集合面板后面板不自动获焦，j/k/Enter 键盘导航失效**
- 面板靠 `tabIndex={0} onKeyDown` 实现 j/k/Enter/x/c 导航（`IssuePanel.tsx:247-269,277`），但挂载时没有任何 `.focus()`/`autoFocus`。
- `openCollection`（`project-store.ts:130`）只切 `view`，不动 DOM 焦点。
- 现象：从左栏点开 Issue 集合后，键盘 j/k 不工作，必须先用鼠标点一下面板。

**P5 — 终端每次 reveal 无条件抢焦点，会把焦点从用户正在操作处夺走（与 P1 行为不一致）**
- `TerminalView` 由 `key={shownSession.id}` 驱动（`RightPane.tsx:187`），换会话即重挂 → reveal → `term.focus()`（`TerminalView.tsx:813`）。
- 现象：切换"有会话的文档"时，重放完成后焦点会在 100–500ms 后突然跳进终端，把光标从左栏/编辑器里拽走。同一个"选中文档"动作，焦点最终落在编辑器还是终端取决于这篇文档有没有会话——行为不可预测（P1 编辑器从不抢、P5 终端总抢）。

**P6 — 没有焦点"归还/恢复"机制**
- 切到无会话的文档时右栏是 launch pad（无终端），没有任何代码把焦点放回编辑器；从 Issue 面板按 Enter 进文档（`IssuePanel.tsx:256`）也不回编辑器。
- 窗口 `blur` 只用于存盘（`DocView.tsx:27-34`）；窗口重新获焦时没有恢复上次焦点位置（仅 `StatusList.tsx:89` 用 window `focus` 去拉 git head，与输入焦点无关）。

**P7 — 焦点动作普遍叠加 `requestAnimationFrame`/`setTimeout`，落点时间不确定（加重 P2/P5）**
- `TerminalView.tsx:813,862` 用 RAF 延后 focus；`CrepeEditor.tsx:112-117 editAction` 两层 `setTimeout`（50ms + 0ms）。
- 这些本是为绕开 Radix 菜单/inert 的焦点抢夺，但叠加重放延迟后，"焦点何时到位"难以预测。

### 一句话根因

焦点逻辑是**挂载耦合**而非**意图驱动**：谁刚挂载/刚 reveal 谁就（在自己的时机里）抢焦点，缺一个"用户这次操作想把焦点放哪、何时放"的中央协调。于是出现 P1/P4 该聚焦却不聚焦、P2/P3 聚焦被异步链路拖延、P5 不该聚焦却被抢——三类毛病同源。

### 可选修复方向（供决策，未实现）

- 引入一个轻量"焦点意图"协调：每次显式导航（selectDoc / openCollection / select session）带上"焦点目标 = 编辑器 | 终端 | 面板"，由目标组件在 ready 后认领，而不是各自 mount 抢。
- P1：让普通 `selectDoc` 也可选把焦点送进编辑器（沿用现有 `focusOnMount` 通道，只是把触发面从"新建"扩到"选中"）。
- P2：把终端 focus 与 inert 解绑——可在容器 reveal 前就允许聚焦（保留"重放期按键不外泄"可改成丢给终端而非吞掉），缩短不可打字窗口。
- P3：先切 UI/焦点再后台 flush（存盘不必挡在切换前；echo 去重已能兜底）。
- P4：集合面板挂载时 `autoFocus`/`.focus()`，让 j/k 立即可用。
- 统一 P1 与 P5 的策略，消除"有没有会话决定焦点落点"的不可预测性。
