## 评估：Iris 对自身开发的帮助有多大（2026-06-12，agent 代写）

结论：帮助是真实的，但分布很不均匀——作为**需求捕获与人-agent 协作媒介**价值最大；作为**上下文注入器**目前很薄；而第一轮的净效率收益大概率为负（补课轮），价值在于它是自己最好的测试场。

### 闭环确实在转

- 本次会话即证据：FOCUS_DOC 指向本 issue，agent 经核心手势被召唤。
- 第一轮：10 个 issue → 7 个提交落地 → 报告归档；issue 内有 agent 写回的「工作记录」，文件即契约的写回纪律真实发生。
- 「第三轮需要处理的问题」是最佳样本：用户口述 → agent 代录整理为 A–J 条需求理解 → 待拍板项清单 → 三批补充逐步收敛 → "需求已闭环"。一场 PRD 谈判完整发生在一份 md 里，git 留痕。这是 issue 状态机 + 聚焦手势 + 写回规约共同诱导出来的，不是裸 markdown 文件夹会自然长出的形态。
- 自产自用：todo 面板/标签/优先级本身来自第一轮 issue「新功能：代办」，落地后即用于管理后续轮次。

### 诚实的折扣

- 第一轮 issue 多数是 Iris 自身缺陷（复制粘贴、滚动条、标题栏、设置页）——工具产生的工单大半是工具自己造成的。早期 dogfood 这是正常且高价值的信号，但意味着目前"省下的时间"还没追上"修自己花的时间"。
- 注入链薄：壳只给一个 FOCUS_DOC 环境变量，让 agent 真正看懂项目的重活仍由手写的 CLAUDE.md + CONVENTIONS 承担。项目自己已意识到（misc 探讨 → 第三轮 A 条 hooks 方案）。
- 协议与 App 难分账：没有 App，`.iris/` + 终端 claude 也能跑约八成流程。App 现阶段的独特增量 = 一键召唤、状态/优先级/todo 可视化、会话↔文档联动、归档灰化。

### 元循环的健康信号

dogfood 在真实反哺协议设计：模板"每任务追加 report"被发现诱导 agent 乱建文件而改规约；状态值大小写需迁移并触发规约修订。协议先行的项目最需要这类反馈，而它只能靠自用获得。
