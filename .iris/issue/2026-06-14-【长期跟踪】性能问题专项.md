目前这个软件的性能还是有很多问题的，总感觉无论干什么都是卡卡的，终端内容的加载尤其慢，有的时候，终端内容切过去的时候甚至会卡掉，你现在开始进行一个全面的检查和调研，然告诉我现在最严重的性能瓶颈是什么

## 2026-06-14 性能调研记录

结论先写：目前最严重、也最符合“终端内容切过去会卡掉”的瓶颈，是终端切换时的完整状态回放链路。每个会话在主进程维护一个 headless xterm 镜像，切换可见会话时渲染进程会销毁旧 `TerminalView`、新建 xterm、向主进程请求 scrollback，然后把主进程序列化出来的最多 5000 行终端状态整体 base64 传到渲染进程，渲染进程再一次性解码成 `Uint8Array`，最后按 16KB 分片 `term.write`。这条链路保护了 alt-buffer/TUI 状态，但它把“切换终端”变成了一次重 CPU、重内存分配、重 IPC、重 xterm parser 的操作；如果终端刚跑过大量输出，或者同时有多个会话，这就是当前最容易造成肉眼卡顿甚至短时间卡死的路径。

关键证据：

- `src/main/session-manager.ts` 为每个 PTY 同步写入 headless xterm，并用 `SerializeAddon` 保存可回放状态；scrollback 配置为 5000 行。
- `getScrollbackForReplay()` 会先 flush 输出、等待 headless parser drain，然后 `addon.serialize({ scrollback: 5000 })`，再把整段 ANSI 转 base64 返回。
- `src/renderer/components/terminal/TerminalView.tsx` 在 `sessionId` 改变时整组件 remount，新建 xterm/WebGL/search/web-links/ResizeObserver/剪贴板等完整实例，再请求 `SESSION_SCROLLBACK`。
- 渲染端拿到 replay 后先 `b64ToBytes(replay.data)` 一次性解码完整回放，再按 16KB 写入 xterm；分片只降低单次 `write` 的阻塞，不能避免完整解码和完整 replay 的总成本。
- 主进程输出聚合使用 8ms batch，但 batch 内追加通过 `Buffer.concat` 复制已有 pending buffer；高吞吐输出时会增加额外内存拷贝。

严重风险清单：

1. 终端切换 replay 风险（最高）
   - 症状：切换终端慢、白屏/隐藏等待、内容多时卡住。
   - 根因：每次切换都完整重建 xterm 并完整回放 scrollback，而不是复用已挂载终端或做增量/懒加载。
   - 放大因素：5000 行 scrollback、base64 传输膨胀、一次性解码、xterm parser 主线程执行、WebGL addon 初始化、React dev StrictMode 额外 mount/unmount。

2. PTY 输出双写和序列化成本
   - 症状：命令大量输出时 UI 变钝，主进程 CPU/内存压力升高。
   - 根因：每个输出 chunk 同时进入 IPC pending buffer 和 headless xterm；headless mirror 需要解析所有 ANSI/TUI 控制序列。
   - 风险：多个活跃会话同时输出时，主进程会成为集中瓶颈；renderer 即使只显示一个终端，也要承受主进程 mirror 的总成本。

3. `.iris` 文件变化后的全量扫描风险
   - 症状：保存文档、agent 批量写 issue/report 后，左栏/中栏有顿挫。
   - 根因：chokidar 150ms debounce 后触发 `PROJECT_SCAN`，扫描会递归读取并解析整个 `.iris` 树的 Markdown、frontmatter 和 todo。
   - 当前数据：本工作区现在约 39 个 Markdown、`.iris` 总文件约 170KB，所以不是当前第一瓶颈；但随着 issue/report 增长到几百篇，这会变成稳定卡顿源。

4. React store 粒度过粗
   - 症状：看似无关的状态变化也会让大块 UI 重新渲染。
   - 根因：`project-store`、`session-store`、`editor-store` 都是 `useSyncExternalStore` 整对象订阅；任意 patch 会通知所有订阅者。
   - 具体影响：session state/title 变化会让左栏 `StatusDot`、右栏 session 列表等重新计算；project scan 替换整棵树后，左栏和中栏集合视图都会重跑。

5. 列表视图无虚拟化、派生计算现场执行
   - 症状：issue/todo/report 列表数量上来后滚动和筛选变慢。
   - 根因：`IssuePanel`、`TodoPanel`、`SimpleList` 每次 render 都从 workspace tree 现场 `collect + filter + sort + map`，没有 memo，也没有虚拟列表。
   - 当前可接受，但规模增长后会和全量扫描叠加。

6. 左栏会话状态点是 N 行重复扫描 sessions
   - 症状：会话多、文档多时左栏刷新变慢。
   - 根因：每个 `DocRow` 内的 `StatusDot` 都订阅整个 session store，并调用 `aggregateDocState(sessions, docPath)` 扫一遍 sessions。
   - 风险：复杂度接近 `文档数 × 会话数`，session 状态频繁变化时尤其浪费。

7. 编辑器更新粒度风险
   - 症状：大文档编辑时输入延迟、切换文档慢。
   - 根因：Crepe/Milkdown 的 `markdownUpdated` 每次更新都会把完整 Markdown 字符串写入 `editorStore.setBody(md)`；store patch 会通知所有编辑器订阅者。
   - 当前文档最大约 25KB，还不算灾难；但长 issue/report 或粘贴大段内容后会明显恶化。

8. 开发环境 StrictMode 放大体感卡顿
   - 症状：dev 模式下比生产包更卡，终端/编辑器初始化看起来被重复执行。
   - 根因：入口使用 `React.StrictMode`，开发环境会额外执行 effect mount/cleanup/mount；终端 replay、Crepe async lifecycle、WebGL 初始化都会被放大。
   - 这不一定代表生产包同等严重，但会影响日常开发判断。

建议优先级：

1. 先处理终端切换 replay：评估缓存/保活最近 N 个 `TerminalView`，或在右栏切换时隐藏而不是销毁；同时把 replay 解码改成流式/分段 base64，避免完整 `Uint8Array` 一次性分配。
2. 给 `SESSION_SCROLLBACK` 增加体积和耗时日志：记录 ANSI 字节数、base64 字节数、serialize 耗时、renderer decode 耗时、term.write 完成耗时，用真实数据确认卡顿阈值。
3. 优化主进程输出聚合：避免 batch 内反复 `Buffer.concat`，改为 buffer list，flush 时一次 concat。
4. 给 session store 增加按 docPath 聚合的派生索引，左栏不要每行重复扫描全部 sessions。
5. `.iris` 扫描先做缓存和增量：至少按 path/mtime/size 缓存 frontmatter/todo 解析结果；文件变化时只重读变更文件，再重建轻量 tree。
6. 集合视图加 `useMemo`，数据量继续增长后再引入虚拟列表。
7. 编辑器 `markdownUpdated` 做 debounce 或 dirty 标记分离：输入过程中不要把完整 Markdown 高频推入全局 store。

## 2026-06-14 性能监视器设计草案

设计理念：先做一个低侵入、默认轻量、专门抓“卡顿因果链”的本地性能监视器。它不是泛泛的漂亮仪表盘，而是用来回答：哪条链路卡、一次卡顿由什么动作触发、卡顿时数据量多大、main/renderer 哪边在忙。

基本原则：

- 默认只保留内存 ring buffer，不自动落盘。
- 不记录终端输出和文档正文，只记录字节数、行数、路径、sessionId、耗时。
- 高频路径只聚合计数，不逐条刷 UI。
- 性能面板最多 1 秒刷新一次，避免监视器自己制造卡顿。
- 生产包可开关，开发包可以展示更详细信息。

指标模型：

- `span`：一段操作的耗时，例如 `terminal.replay.serialize`、`terminal.replay.decode`、`terminal.replay.write`、`project.scan`、`editor.crepe.create`、`ipc.invoke`。
- `counter`：高频计数，例如 `pty.bytesPerSecond`、`session.outputBatches`、`markdownUpdated.count`。
- `gauge`：当前状态，例如 renderer heap、main heap、event loop lag、active sessions、最近一次 replay bytes。

记录结构示意：

```ts
{
  ts,
  process: 'main' | 'renderer',
  name: 'terminal.replay.write',
  durationMs: 184,
  tags: {
    sessionId,
    bytes: 734000,
    chunks: 45,
    scrollbackLines: 5000
  }
}
```

第一阶段优先埋点：

1. 终端切换 replay
   - main serialize 耗时。
   - replay ANSI 字节数、base64 字节数。
   - renderer 等待 `SESSION_SCROLLBACK` 的总耗时。
   - base64 decode 耗时。
   - xterm 分片写入总耗时。
   - 从切换 session 到 terminal reveal 的总耗时。

2. PTY 输出
   - 每秒输出字节数。
   - 8ms batch 实际 batch 数。
   - batch 最大字节数。
   - pending buffer flush 大小。
   - headless xterm write 压力。

3. `.iris` 扫描
   - scan 总耗时。
   - Markdown 文件数。
   - 总读取字节数。
   - frontmatter/todo 解析耗时。
   - watcher 触发原因和 batch 数。

4. Renderer 卡顿
   - Long Task（超过 50ms 的主线程阻塞）。
   - 近似 FPS / dropped frame。
   - LeftPane、MiddlePane、RightPane、TerminalView、IssuePanel 等大组件 render 次数。
   - renderer heap 使用量。

5. 编辑器
   - Crepe create/destroy 耗时。
   - `markdownUpdated` 频率。
   - 每次传入 store 的 Markdown 长度。
   - save 耗时。

6. IPC
   - 每个 `window.api.invoke` 的 channel、耗时、payload 大小、response 大小。
   - 重点关注 `SESSION_SCROLLBACK`、`PROJECT_SCAN`、`DOC_READ`、`DOC_WRITE`。

UI 方案：

- 做一个“性能面板/开发者抽屉”，而不是先做复杂 profiler。
- 顶部显示最近 60 秒最大 long task、main/renderer 内存、event loop lag。
- 慢操作排行显示最近最慢的 20 个 span。
- 终端页签展示每次 session 切换的 replay 瀑布：serialize、IPC、decode、write、reveal。
- 扫描页签展示每次 `.iris` scan 的文件数、字节数、耗时。
- IPC 页签展示慢 IPC 排行。
- 提供一键导出 NDJSON/JSON，便于贴回 issue。

建议落地顺序：

1. 建最小 perf runtime：`span/counter/gauge + ring buffer + export`。
2. 先只接入终端 replay 和 `PROJECT_SCAN` 两条链路。
3. 加简单性能面板，能看慢操作排行和终端 replay 瀑布即可。
4. 再补 IPC wrapper、Long Task、编辑器和 React render 计数。
5. 根据真实数据决定后续优化顺序，避免继续靠体感猜瓶颈。

## 2026-06-14 低概率终端切换后未加载问题

现象：切换到某个终端后，极低概率出现“内容没有加载成功”的状态，视觉上像空白/未完成 replay。

本次排查结论：最可疑且已修复的根因是 replay 的 seq 边界竞态。原协议是 renderer 先订阅 live output，再向 main 请求 `SESSION_SCROLLBACK`，main 返回 `{ data, lastSeq }`，renderer 写入 replay 后只补写 `seq > lastSeq` 的 holdQueue。问题在于 main 之前返回的 `lastSeq` 是函数末尾读取的 `managed.scrollbackLastSeq`，它可能已经包含了“replay 边界之后才到达”的输出。

具体竞态：

1. main flush 当前 pending emit。
2. main 等 headless xterm 的 `write('', cb)` drain fence，再 serialize 快照。
3. 在 fence 等待、serialize、base64 转换这段时间里，PTY 可能又输出一批 TUI redraw。
4. 这批输出会被正常 emit，renderer 因为已经订阅，会先放进 holdQueue。
5. main 函数末尾读取到更靠后的 `scrollbackLastSeq` 并返回。
6. renderer 认为 holdQueue 中 `seq <= lastSeq` 的输出已经包含在 replay 里，于是丢掉。
7. 如果被丢的是清屏后的 redraw，就会表现为切换后内容空白/像没加载成功。

已做修复：

- `src/main/session-manager.ts`
  - 在 `getScrollbackForReplay()` 内，先清掉 pending emit timer 并 flush 当前 pending emit。
  - 随后立即冻结 `replayLastSeq = managed.scrollbackLastSeq`。
  - 再插入 headless parser fence 并 serialize。
  - 返回冻结的 `replayLastSeq`，而不是函数末尾可能已经前进的 `scrollbackLastSeq`。
  - 含义：冻结点之后到达的输出一律视为 live 补写，renderer 不会误丢。

- `src/renderer/components/terminal/TerminalView.tsx`
  - 给 `SESSION_SCROLLBACK` 增加 5 秒 failsafe timeout：如果 replay IPC 长时间不返回，进入 catch 路径，至少 flush holdQueue 并 reveal 终端，不再永久 `visibility:hidden`。
  - 给 replay fence 的 `term.write('', cb)` 增加 5 秒 failsafe：如果 xterm drain 回调极低概率不触发，也会降级 reveal，并打印 warning。

验证：

- `npm run typecheck` 通过。

后续仍需监控：

- 性能监视器应记录 `SESSION_SCROLLBACK` 超时、replay fence 超时、replay lastSeq、holdQueue 最大长度、被补写/被过滤的 batch 数。
- 如果仍出现空白，需要区分是 replay 数据本身为空、xterm renderer/WebGL 绘制失败，还是 TUI 在 resize 后没有主动重绘。
