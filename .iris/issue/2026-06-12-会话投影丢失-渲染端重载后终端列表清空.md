# 会话投影丢失：渲染端重载后终端列表清空

## 现象

agent 在修改渲染端代码的过程中（dev 模式），右栏所有终端突然消失；但检查发现 PTY 进程并没有被结束，仍在后台运行。会话不可见、不可切换、不可关闭——变成"幽灵会话"。

## 根因（已定位，证据确凿）

**渲染端的会话投影（session-store）只靠增量事件维护，没有任何全量恢复路径。**

主进程才是会话池的真相之源（`src/main/session-manager.ts` 的 `Map<string, ManagedSession>`），渲染端 `src/renderer/stores/session-store.ts` 只是它的投影。这个投影的数据来源只有两个：

1. `session.open` 指令的 commit（`sessionStore.handleCreated`）；
2. `evt:session:*` 增量广播（state-changed / exited / destroyed，经中断 ISR 进 `handlePatch` / `handleDestroyed`）。

一旦渲染进程重载，模块级 `state` 重新初始化为空数组，而上述两个来源都不会回放历史——投影从此与主进程永久脱节。

三个本应兜底的东西都是断的：

- `CHANNELS.SESSION_LIST`（`session:list`）查询通道**主进程已实现**（`src/main/ipc.ts:205` → `sessionManager.list()`），但**渲染端没有任何调用者**——全仓 grep 零命中。
- `sessionStore.reset(sessions)` 方法存在（注释写明是给"项目切换"用的），但**没有任何调用者**——死代码。
- `project.open` 的 commit（`src/renderer/cpu/isa/project-isa.ts`）只调 `projectStore.handleOpened`，从不触碰 session-store。

启动序列（`src/renderer/index.tsx` 的 `bootstrap()`）只做 `initSettingsStore` + `wireInterrupts` + 渲染 + 重开 lastRoot，同样没有会话水合步骤。

## 触发条件

任何导致渲染进程重载、而主进程不动的事件：

1. **dev 模式下改渲染端代码**（本次观察到的场景）：Vite HMR 链一旦失效（模块级 store、入口文件等不可热替换的模块被改动）就退化为整页 full reload。agent 编辑 renderer 文件时这是常态。
2. **手动刷新**：Ctrl+R / F5——生产构建同样中招，不是 dev 独有。
3. **渲染进程崩溃后恢复**：Electron 重建 webContents，同样从空投影开始。

注意区分：dev 模式下改**主进程**代码会让 electron-vite 重启整个 Electron，PTY 真的死掉——那是另一回事（dev 固有代价）。本 issue 针对的是"主进程活着、渲染端失忆"。

## 影响

- 会话不可见、不可关闭，PTY/ConPTY 进程持续占资源；挂着的 agent 还在干活但用户再也看不到它的输出入口。
- 新加的退出确认（confirmOnQuit）统计的是**主进程**的会话数——用户会看到"仍有 N 个会话在运行"却在 UI 里一个都找不到，困惑加倍。
- 左栏状态点（aggregateDocState）同样基于这份投影，全部熄灭——注意力调度面板失明。

## 为什么内容可以完整恢复

主进程为每个会话维护 headless 镜像（5000 行 scrollback，Marina 状态回放协议），TerminalView 挂载时本来就走"订阅→拉全量回放→续流"的链路。**所以丢的只是生命周期清单，不是终端内容**——只要把清单水合回来，点开任何会话都能完整还原画面。修复成本低、收益完整。

## 修复方案（建议）

三层，按必要性排序：

1. **启动水合（必做）**：`bootstrap()` 里（`initSettingsStore` 之后）调 `SESSION_LIST`，`sessionStore.reset(list)`。重载后投影立即与主进程对齐。
2. **project.open 接通 reset（必做，顺手修掉死代码）**：commit 里重新拉 `SESSION_LIST` 并 reset。同时解决注释里写了却从未实现的"项目切换后投影重置"。跨项目会话（`SessionInfo.projectRoot` 与当前项目不符）如何呈现需要拍板（见下）。
3. **自愈兜底（建议做，便宜）**：中断 ISR 收到**未知 sessionId** 的 `evt:session:*` 事件时，触发一次全量 re-list。这样即使未来出现新的脱节路径（漏发事件、竞态），投影也会在下一个事件到来时自动归队，而不是永久错下去。与"ISR 内只用确定性状态比对"的纪律一致：未知 id 就是确定性的脱节证据。

## 待拍板

1. ~~跨项目幽灵会话的呈现~~——用户裁定：暂无多项目问题，不处理。
2. 水合时 `activeSessionId` 的选择：复用现有规则（取列表最后一个 = 最新创建）。

## 实现记录（2026-06-12，三层方案全部落地）

1. **启动水合**：`session-store.ts` 新增 `hydrateSessions()`（拉 `session:list` → `sessionStore.reset(snapshot)`）；`index.tsx` 的 `bootstrap()` 在 `wireInterrupts()` 之后、首次渲染之前 `await` 它——重载后的第一帧就能看到幸存会话。
2. **project.open 接通**：`project-isa.ts` 的 commit 在 `projectStore.handleOpened` 之后调 `hydrateSessions()`，原 `reset()` 死代码从此有了真实调用方。单项目模型下全量快照替换即正解。
3. **ISR 自愈兜底**：`interrupts.ts` 的 session-projection ISR 里，`state-changed` / `destroyed` 事件若携带投影中不存在的 sessionId，视为确定性脱节证据，触发一次全量 re-list 而不是对空打补丁。纯状态比对，零启发式，符合 ISR 纪律。

终端画面恢复不需要额外工作：TerminalView 挂载本就从主进程 headless 镜像全量回放。

验证：typecheck 三进程通过，34 个单测通过。手工验收建议：dev 模式开 2-3 个会话 → 按 Ctrl+R 强制重载 → 列表应完整回来，点开任一会话画面完整、可继续输入。
