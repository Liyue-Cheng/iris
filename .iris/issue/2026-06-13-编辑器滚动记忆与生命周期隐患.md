从「第四批」里单独摘出来的一条，因为它牵出的不只是一个滚动 bug，而是
**Crepe（Milkdown）编辑器异步生命周期**这一层的系统性脆弱。滚动记忆本身已修
复并可用，但背后的隐患值得单独记录、日后审视。

基线：未提交，工作树基于 `fda7d1e`。涉及编辑器正文区（中栏）。

***

## 1. 现象

需求：切走文档再切回，正文应停在原阅读位置（第四批 C1/C2）。

实现过程踩了三个坑，逐个暴露出更深的问题：

1. **完全无效**（第一版）：滚动位置存了等于没存。
2. **棘轮式上漂**（第二版）：拉到底 → 切走再切回不回底部，而是**往上窜一截**；
   再切再窜，约三次就回到最顶端。偶尔又恰好正常。
3. **再次完全失效**（第三版）：为修棘轮换了选元素的逻辑，结果存取彻底断链。

最终第四版修好。下面把每一步的根因讲透——因为这些根因就是「隐患」的来源。

***

## 2. 为什么编辑器这一层特别容易出错

四个结构性事实（已落到 `docs/milkdown-crepe-api.md`）：

* **没有滚动 API**。Crepe / Milkdown / ProseMirror 都不提供"保存/恢复滚动位置"
  的接口。唯一的正经句柄是 `editorViewCtx → EditorView → view.dom`
  （`.ProseMirror` 节点），之后全是裸 DOM 操作。

* **真正的滚动容器由主题 CSS 决定，且不在固定那一层**。本仓库里 Crepe 自己不建
  滚动容器（其主题 CSS 没有任何 `overflow:auto/scroll` 落在正文层），滚动来自我们
  `global.css` 给 `.crepe-host` 的 `overflow-y-auto`；而 `.crepe-host .milkdown`
  又被钉成 `height:100%`。"哪个元素 `scrollTop` 会变"必须运行时判定，硬编码任何
  一层都会错。

* **挂载是异步的，且分多帧排版**。`crepe.create()` 是 Promise，resolve 之后内容
  仍会在随后若干帧里继续排版，`scrollHeight` 持续变化。任何"挂载即恢复"都会赶在
  布局完成之前。

* **`scrollTop`** **的写也会触发** **`scroll`** **事件**，浏览器不区分人滚还是代码设。这点和
  上一条叠加，正是棘轮的温床。

***

## 3. 根因分析

### 3.1 棘轮（第二版的核心 bug）

第二版：捕获阶段监听 `scroll` 持续保存 + 挂载后定时重试写 `scrollTop` 恢复。两者
**没有隔离**，于是：

```
底部 = 1000，保存 Map=1000
切回 → 恢复开始：第 1 帧内容只排了一部分，maxScroll 仅 ~700
  代码 scrollTop=1000 → 被夹紧到 700
  这次"设置"触发 scroll 事件 → 保存监听抓到 → Map=700   ← 好值被自己覆盖
若这一轮在真正长到 1000 之前就落定/超时 → 最后存进去的是某个 < 1000 的夹紧值
```

* **单调上漂**：恢复写回的永远是被夹紧的值（≤ 目标），只降不升 = 棘轮。

* **每次约一屏**：首帧能排出的高度大致恒定，所以每次少掉的量也恒定 ≈ 一个视口。

* **偶尔正常**：某次排版够快、在落定前真的摸到 1000，那一帧最后存的就是 1000。

一句话：**保存逻辑分不清"用户滚动"和"恢复时自己设的 scrollTop"，把过渡态也持久
化了。**

### 3.2 选错滚动元素（第三版的"完全失效"）

为修棘轮，第三版把"找滚动层"改成要求 `getComputedStyle(el).overflowY ∈ {auto, scroll}` 且 `e.target === scroller` 严格相等。真正能滚的那层未必满足这套苛刻条件
（或解析返回 null），于是保存条件永不成立 → Map 永远空 → recall 恒为 0 → 不恢复。

**教训**：第二版能产生棘轮，恰恰证明它的 `contains` 选元素 + 链式写入是"能滚、能
存、能取"的；修 bug 时不该顺手替换掉这套被验证可用的机制。

***

## 4. 最终方案（第四版，已落地）

把滚动记忆收敛进一个自洽的 keeper：`src/renderer/lib/scroll-memory.ts` 的
`attachScrollMemory({ key, content })`。要点：

1. **恢复期间禁存（`restoring`** **闸）**——根治棘轮。恢复时发生的所有 `scroll`（全是
   程序化的）一律不写 Map，所以持久化的永远是用户真实位置。
2. **保存用** **`t.contains(content)`** **认滚动层**——`content` 是内容节点（Crepe 的
   `view.dom` / CodeMirror 的 `view.scrollDOM`）。真正的滚动容器是 content 的祖先，
   `contains` 成立；代码块内部的横向滚动条在 content 内部，`contains` 不成立，自动
   排除。不预设是哪一层。
3. **恢复用链式写入**——从 content 向上 8 跳，对每个 `scrollHeight>clientHeight`
   的祖先写 `scrollTop=目标`；真正的滚动层接住，其余无害 no-op。
4. **靠"高度稳定"判定恢复完成**——`scrollHeight` 连续 3 帧不变即认为排版完成，结束
   恢复、放开 `restoring` 闸；并设 120 帧（\~2s）上限兜底，绝不空转。
5. **生命周期安全**——`stop()` 置 `stopped` 让僵尸 rAF 立即退出；中途被销毁不回存
   半截位置。

净效果：**只有用户真实滚动会被持久化**，恢复自己制造的滚动一律不进 Map，单调下降的
棘轮在机制上不可能再发生。

### 涉及文件

* `src/renderer/lib/scroll-memory.ts` —— 新增，keeper 实现（Map + `attachScrollMemory`）。

* `src/renderer/components/doc/CrepeEditor.tsx` —— `create().then()` 里用
  `editorViewCtx` 取 `view.dom`，挂 keeper；卸载 `keeper.stop()`。

* `src/renderer/components/doc/SourceEditor.tsx` —— CodeMirror 用 `view.scrollDOM`
  作 content，挂同一个 keeper。

* `docs/milkdown-crepe-api.md` —— 新增，按本机 `.d.ts` 提取的 Crepe/Milkdown API
  速查（含"没有滚动 API、该怎么做"的结论）。

## 5. 仍存在的隐患（这条 issue 的真正价值）

滚动记忆好了，但编辑器这层的脆弱没有消除。按风险排序：

1. **StrictMode 下 Crepe 双挂载竞态（dev）**。`index.tsx` 用了 `<React.StrictMode>`，
   dev 下 effect 跑 setup→cleanup→setup。CrepeEditor 因此会"建实例#1 → 立刻
   destroy#1 → 建实例#2"，而 `create()/destroy()` 都是 async 且不互相 await，两个
   实例可能同时操作同一个 `root` DOM，残留 `.milkdown`、泄漏监听/observer。生产构建
   剥离 StrictMode 后不复现，但**说明我们对"上一个没销毁完就建下一个"没有任何防护**。
   建议：给 CrepeEditor 加挂载序列化（destroy 完成后再 create，或忽略过期的 then）。

2. **`create()/destroy()`** **异步未串行化**。快速连点不同文档时，`doc.save` 是按
   `doc:{path}` 串行的，但**编辑器的建/毁没有同等保证**。极端时序下可能出现 then()
   写入已被替换的会话。目前靠 `destroyed` 闭包变量挡，覆盖面有限。

3. **恢复是启发式而非事件驱动**。"高度连续 3 帧不变 = 排版完成"对图片/代码高亮等
   **异步/懒加载内容**可能误判——内容在恢复结束后才变高/变矮，会让位置略偏；若变矮
   触发夹紧且此刻 `restoring` 已放开，会发生一次性（非棘轮）回退。理想方案是监听
   Crepe `mounted` + 内容区 `ResizeObserver` 稳定信号，而不是数帧。

4. **全局** **`document`** **捕获监听**。每个 keeper 在 `document` 上挂一个捕获 `scroll`。
   当前同时只挂一个编辑器，没问题；若未来出现多编辑器并存（分屏/预览），需要确保
   `content` 过滤足够隔离，否则互相串扰。

5. **耦合 Milkdown 内部**。`editorViewCtx`、`view.dom` 是 milkdown 的内部约定，
   升级版本若改 DOM 结构或 ctx 形状会静默失效。已在 `docs/milkdown-crepe-api.md`
   备注，升级时需回归。

6. **滚动记忆仅存内存**。`scroll-memory.ts` 的 `Map` 是模块级、进程内、跨应用重启即
   丢（设计如此，但与第四批 B3"关窗 flush"不同层级，重启不保留阅读位置）。容量无界
   但每条仅一个数字，可忽略。

7. **`scrollTop`** **写仍会发** **`scroll`**。`restoring` 闸是时间窗口性的；窗口外的程序化
   滚动（理论上不该有，但第三方插件可能触发）仍会被记。属低概率。

## 6. 复现与验收

复现（第二版棘轮，现已修）：

1. 打开一篇够长的文档（正文需可滚动）。
2. 拉到最底。
3. 切到另一文档/集合视图，再切回。
4. 旧行为：位置上窜一截；连续三次到顶。新行为：稳定停在底部。

验收：

* [x] 拉到底来回切多次，位置稳定不漂。

* [x] 拉到中部来回切，停在中部。

* [x] 源码模式（CodeMirror）同样保留位置。

* [x] 切换够快（恢复未完成就切走）不破坏下次恢复。

* [x] 长文档（含代码块/图片）位置基本正确（允许极小偏差，见隐患 3）。

## 8. 追加处理：Crepe 生命周期串行化（2026-06-14）

### 我的判断

滚动记忆本身已经由 `scroll-memory.ts` 的 `restoring` 闸修住了；继续追这条 issue 时，
真正应该先处理的是第 5 节里的前两个高风险项：**StrictMode 双挂载竞态**和
**`create()/destroy()`** **异步未串行化**。

原因是：滚动恢复的启发式失败，多数表现为位置略偏；但 Crepe 生命周期竞态一旦撞上，
可能是同一个 root DOM 上同时存在两个异步实例，表现会更散：残留 `.milkdown`、监听泄漏、
旧实例晚到后写 baseline、滚动 keeper 绑到已过期 DOM，甚至编辑器空白。也就是说，滚动
bug 是症状，`create/destroy` 没有所有权边界才是更底层的风险。

### 我做了什么

新增并接入了一个专门的生命周期 helper：

* `src/renderer/lib/crepe-lifecycle.ts`

  * 新增 `mountCrepeSerially({ root, crepe, onCreated, label })`。

  * 用 `WeakMap<HTMLElement, Promise<void>>` 按 root DOM 排队。

  * 同一个 root 上，前一个 Crepe 实例必须停止并完成 `destroy()`，后一个实例才允许
    `create()`。

* `src/renderer/components/doc/CrepeEditor.tsx`

  * 不再直接 `void crepe.create().then(...)` / cleanup 里 `void crepe.destroy()`。

  * 改为通过 `mountCrepeSerially` 挂载。

  * `onCreated` 里再做 `setBodyBaseline()`、取 `editorViewCtx`、绑定
    `attachScrollMemory()`。

  * cleanup 里先 `keeper.stop()`，再 `lifecycle.stop()`。

* `src/renderer/components/doc/RootReadme.tsx`

  * 只读 README 的 Crepe 也走同一套串行化，避免根 README 视图在 StrictMode/快速切换
    下成为另一个生命周期漏点。

* `docs/milkdown-crepe-api.md`

  * 把第 6 节里过时的“向上找第一个 overflow-y:auto/scroll 并监听该容器”建议，改成
    当前实际方案：`attachScrollMemory({ key, content })`、document 捕获保存、
    祖先链写入恢复、恢复期间禁存。

* `src/renderer/lib/crepe-lifecycle.test.ts`

  * 新增单元测试覆盖两个关键时序：

    1. 同一个 root 上第二个 Crepe 不会在第一个 `destroy()` 完成前 `create()`。
    2. `create()` 已启动但 cleanup 先到时，`create()` resolve 后会立即 `destroy()`，
       且不会执行 `onCreated`。

* `vitest.config.ts`

  * 测试收集范围加入 `src/renderer/lib/**/*.test.ts`，并补上 `@renderer` alias。

### 实现方式

`mountCrepeSerially` 的关键不是只串 `create()`，而是把**整个 mounted lifetime**放进队列：

1. 读取当前 root 的上一条 lifecycle Promise。
2. 等上一条结束。
3. 如果当前 effect 已经 cleanup，则直接跳过，不创建 Crepe。
4. 否则 `await crepe.create()`。
5. 如果创建后仍未停止，执行 `onCreated()`，并等待 `stop()`。
6. 最后只要 `create()` 成功过，就 `await crepe.destroy()`。
7. 如果当前 Promise 仍是 root 的最新队列项，完成后从 WeakMap 删除。

这样覆盖了三个关键竞态：

* **StrictMode setup→cleanup→setup**：第二次 setup 会排在第一次 destroy 后面。

* **快速切文档/切视图**：旧实例的 async create 即使晚到，也不会再执行 `onCreated()`；
  它只会完成后立刻 destroy。

* **destroy 慢于下一次 create**：下一次 create 会等待上一轮 destroy 完成，避免两个
  Crepe 同时操作同一个 root DOM。

这次处理后，第 5 节中的 1、2 两项已由机制层面收敛；第 3 项“滚动恢复仍是高度稳定
启发式”仍然存在，后续如果文档里图片/异步高亮导致明显偏移，再考虑引入
`ResizeObserver` 或更细的 mounted/resize 稳定信号。
