这玩意的滚动条经常出现，消失，而且没有应用主题，与此同时，这个和粗大进度条出现和消失的时候，会导致文本本身的抖动，这不是我希望的，你现在开始给一个实现方案

## 分析（agent，2026-06-12）

三个症状是一个根源加一个缺失：

1. **时隐时现**：编辑器滚动容器是 `CrepeEditor.tsx` 上的 `overflow-y-auto`。内容高度在视口阈值附近来回跨越（打字增删行、切换长短文档、Crepe 浮层撑高内容）时，滚动条就反复出现/消失。
2. **文本抖动**：Windows 上 Chromium 经典滚动条占约 15px 布局宽度，出现/消失会改变容器内容区宽度；而 `.ProseMirror` 是 `max-width: 48rem; margin: 0 auto` 的居中排版（`global.css`），内容区宽度一变，整列文字水平平移——这是抖动的直接机制。
3. **没主题、粗大**：`global.css` 中没有任何 `::-webkit-scrollbar` 或 `scrollbar-color` 规则，全 app（编辑器、左栏、issue 面板、设置页、右栏）都在用 Chromium 原生滚动条，仅靠 `color-scheme` 区分明暗，与 Rose Pine 不符。

环境前提：Electron 31 = Chromium 126，`scrollbar-gutter`（Chrome 94+）与 `::-webkit-scrollbar` 均可用。

## 实现方案

改动只落在 `src/renderer/styles/global.css`，两步：

1. **治抖动 —— gutter 常驻**：给 `.crepe-host` 加 `scrollbar-gutter: stable both-edges`。无论滚动条是否出现都预留等宽空间，内容区宽度恒定，文本不再平移；`both-edges` 两侧对称预留，保证 48rem 正文列严格居中（单边 `stable` 会让正文偏移半个滚动条宽度）。
2. **治丑和粗 —— 全局主题化滚动条**：加一组全局 `::-webkit-scrollbar` 规则映射到主题 token，全 app 统一生效（含 xterm 的 `.xterm-viewport`），三个 Rose Pine 变体经 var 间接自动跟随主题：
   - 宽度约 10px；track 透明；
   - thumb 用 `var(--rp-highlight-med)`，hover 升到 `var(--rp-highlight-high)`，配 `background-clip: content-box` + 透明 border 做出细条视觉（实际命中区域不缩水）；
   - corner 透明。

不采用标准属性 `scrollbar-width: thin` + `scrollbar-color` 的原因：一旦设置会使 `::-webkit-scrollbar` 整组失效，且无 hover 态、无圆角，质感达不到 Rose Pine 要求。

## 验收点

- 长/短文档来回切换，正文列不发生水平移动。
- 持续输入使文档从不满屏变为满屏，滚动条出现瞬间正文无抖动。
- 三个主题变体下滚动条颜色随主题切换；左栏、issue 面板、设置页、终端滚动条样式一致。

## 实现记录（agent，2026-06-12）

已按方案落地，改动仅 `src/renderer/styles/global.css` 新增一节：

- `.crepe-host { scrollbar-gutter: stable both-edges }` —— 治抖动；
- 全局 `::-webkit-scrollbar` 系列规则：10px 宽、track/corner 透明、thumb 用 `var(--rp-highlight-med)`（hover 升 `--rp-highlight-high`），`background-clip: content-box` + 2px 透明 border 做出 6px 细条视觉。

核对过 xterm 自带 CSS：其 vscode 风格自定义滚动条（`.xterm-scrollbar`）默认模式不启用，实际滚动条在 `.xterm-viewport` 上为原生 webkit 滚动条，全局规则可覆盖。

状态置 `in_progress`：代码已完成，等用户按上方验收点手工验收后置 `done`。

## 终端滚动条同步（agent，2026-06-12，第二轮）

上一轮"全局 webkit 规则可覆盖 xterm"的判断**有误**：项目装的是 xterm
**6.1.0-beta**，6.x 已把视口滚动条重写为 vscode 风格的 DOM 自定义滚动条
（`.xterm-scrollable-element > .xterm-scrollbar > .xterm-slider`），
`::-webkit-scrollbar` 伪元素够不到它，故终端与编辑器外观不一致。本轮同步：

- `theme/xterm-themes.ts`：每个主题新增 `scrollbarSliderBackground` /
  `Hover` / `Active`（= highlight-med / highlight-high / muted 字面量，与
  CSS token 手工对齐）及 `overviewRulerBorder` = 背景色（设 scrollbar.width
  会顺带启用 overview ruler，其常画的 1px 边线用背景色隐形）。
- `TerminalView.tsx`：Terminal 选项加 `scrollbar: { width: 10 }`，与全局
  10px 对齐。
- `global.css`：给 `.xterm-slider` 补 2px 透明 border + 5px 圆角 +
  `background-clip: content-box !important`（xterm 运行时注入的
  `background` 简写会重置 clip，必须 important），呈现与编辑器一致的
  6px 细条；并修正上一轮注释中的错误说法。

已知差异（设计使然，不再追平）：终端滚动条是 vscode 式覆盖层，滚动/悬停
时显示、闲置淡出（xterm 写死 `ScrollbarVisibility.Auto`），不占布局故无
抖动问题；编辑器侧为常驻原生滚动条。

新增验收点：终端滚动时滑块颜色/粗细/圆角与编辑器一致，hover 变亮；三主题
跟随；滚动条左侧无 1px 异色竖线（overview ruler 边线已隐形）。`npm run
typecheck` 通过。
