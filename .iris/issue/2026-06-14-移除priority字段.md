把软件对 issue `priority` 字段的**全部支持一次性删除**：不再有优先级排序、分组、行内图标、头部编辑器、批量改优先级。`priority:` 从此只是一个**软的、被 App 忽略的 frontmatter 键**——写不写、写什么，App 都不再特殊对待。

> 注：这里删的是「issue 优先级」这一产品概念。front-cpu 指令元数据里的 `InstructionMeta.priority`（`*-isa.ts` 里的 `priority: 5`）是另一回事，与本 issue 无关，**未动**。

## 一、为什么删（第一性原理）

priority 是 round-5 抄 Linear 时引入的，但它从来不是 Iris 的协议概念——`software-definition.md` 和 `.iris/CONVENTIONS.md` 都**没有定义过 priority**。CONVENTIONS 明确定义的只有 `status:`（软状态机）与 `reflects:`。定义书提 issue 面板只说「Linear **风格**」，指的是形态（密度、分组、看板），不是 Linear 的优先级模型。

更根本的，从第一性原理看，priority 这个行政功能在 AI 时代不该存在：

1. **它的本质是「为稀缺的人力执行排队」。** 优先级、故事点、燃尽图同属一类——配给和协调他人劳动的工具。前提是执行慢、贵、少，所以要排序、固化、防反复争。它是组织的管理开销，不是工作本身。

2. **AI 时代抽掉了这个前提——稀缺性反转了。** agent 便宜、可并行、随叫随到，执行不再是瓶颈。稀缺的东西从「谁来干」变成「干什么值得、干得对不对」：从执行变成判断。而 priority 配给的恰恰是执行容量——它在给一个不再稀缺的资源排队。真正稀缺的判断力/注意力，由「**此刻哪个 agent 卡着等你**」分配（左栏会话状态点），不是靠一个静态 rank。

3. **它是一份手工维护的缓存，而 AI 让被缓存的计算变廉价了。** priority 缓存的是「接下来做哪个」这个决策的答案，冻成枚举省得重算。但缓存会腐烂（所有 PM 工具的优先级最后都变成「全是 P1」、没人更新、互相博弈）。而「读完所有 active issue、按当前现实排个序」现在随口问 agent 就能得到，读的是真实内容，强一个量级。**当重算变廉价，缓存只剩腐烂和维护成本，是纯负债。**

Iris 处理「接下来做什么」的原生方式是三件**事实**，没有一件是 priority：`status`（是否 active）+ 会话状态（此刻有没有 agent 等你）+ 按需让 agent 排序（产出是一篇 report 快照/文件，而非常驻字段）。

人偶有 repo 里没有的私有上下文（「客户在发火」「周五要演示」）——那是**内容**，写进正文或当个 label 即可，不需要 App 为它造排序引擎。「键硬值软」已经兜住：想写 `priority: 救命` 随你写，App 只是不再为它做任何机制。

## 二、改了什么（changelog，已 typecheck + build 通过）

* `src/renderer/lib/doc-utils.ts` — 删除 `SOFT_PRIORITIES`、`docPriority()`、`priorityRank()`。

* `src/renderer/components/collection/parts/PriorityIcon.tsx` — **整文件删除**（Linear 式条形图标引擎）。

* `src/renderer/components/collection/parts/DisplayMenu.tsx` — `GroupBy`/`SortBy` 去掉 `'priority'`，对应 label 与默认 `groupOptions` 同步。

* `src/renderer/components/collection/IssuePanel.tsx` — 删 `PriorityEditor`、`PRIORITY_CLS`、`PriorityIcon` 引用；移除优先级**排序/分组/段头 glyph**、行首优先级图标、批量条「设优先级」；GRID 列轨道去掉优先级列；默认排序由 `priority` 改为 `date desc`。

* `src/renderer/components/doc/TypedHeader.tsx` — 删单篇头部 Row 2 的优先级输入框 + 候选下拉（仅保留 LabelsEditor）；清理 `SOFT_PRIORITIES`、`ChevronDown` 等失效 import。

* `src/renderer/lib/issue-actions.ts` — 注释更新（`setDocField` 仍是通用助手，`setDocStatus` 内部仍用）。

* `src/renderer/lib/view-prefs.ts` — localStorage KEY `v1`→`v2`，废掉旧的 `priority` 分组/排序偏好，防止陈旧值命中已删除分支导致列表行消失。

## 三、刻意没做的事

* **未批量重写既有文档的 frontmatter。** 现存 issue 里残留的 `priority: medium` 等行从此是惰性软值，App 直接忽略，无害。批量清洗属于数据层操作、且违背「不无故改数据文档」，留作可选的后续动作——若要做，再单开。

* **未在 CONVENTIONS 里加任何关于 priority 的话。** 它本就不是协议概念，删掉即回到协议原状。
