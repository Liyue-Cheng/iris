## 现象

本项目 dogfood 时，Claude 能读到根目录 `CLAUDE.md`，但 Codex 没有自动读到任何项目级 Iris 引导。实际检查结果：

- 仓库根目录存在 `CLAUDE.md`。
- 仓库根目录不存在 `AGENTS.md`。
- 当前 `CLAUDE.md` 里包含 Iris 项目管理入口说明、`.iris/CONVENTIONS.md` 读取要求、`FOCUS_DOC` 协议和权威文档列表。

这说明 Iris 当前没有把“项目级 agent 入口文件”管理到一个跨 agent 可用的事实状态。至少在本仓库里，Iris 的协议入口与真实入口已经分裂：Claude 走 `CLAUDE.md`，Codex 侧没有对应入口。

## 已确认的实现状态

- `software-definition.md` §3 和附录 A 把根 `AGENTS.md` 定义为项目级标准入口：Iris 只追加一段引导。
- `roadmap.md` M5 冷启动验收要求 init 向导生成 `.iris/`，并向根 `AGENTS.md` 追加引导段。
- `src/main/project-manager.ts` 的 `initIris()` 只创建/追加 `AGENTS.md`，不检测、不迁移、不提示已有 `CLAUDE.md`。
- `src/main/iris-templates.ts` 只维护 `AGENTS_GUIDANCE` / `AGENTS_GUIDANCE_MARKER`，没有 Claude/Codex 等入口文件兼容策略。
- `src/main/project-init.test.ts` 只覆盖 `AGENTS.md` 创建、追加、幂等，不覆盖“已有 `CLAUDE.md` 但缺少 `AGENTS.md`”的场景。

## 问题判断

这是协议/实现/自用状态之间的断裂，不只是本机配置问题：

1. 协议上说标准入口是 `AGENTS.md`。
2. 实现上 init 只会维护 `AGENTS.md`。
3. dogfood 仓库实际只有 `CLAUDE.md`，没有 `AGENTS.md`。
4. 因此支持 `AGENTS.md` 的 agent（例如 Codex）无法从项目根自动进入 Iris 协议，只能靠 SessionStart focus-context 或用户手动提示降级。

这会直接削弱 M3/M5 的验收目标：“agent 不经任何手动粘贴即正确回答当前聚焦文档和工作区”。`FOCUS_DOC` 只告诉它焦点路径；“先读 `.iris/CONVENTIONS.md`、必要时读 `~/.iris/CONVENTIONS.md`、不要修改宪法”等规则仍需要入口文件或 hook 内容承担。

## 建议方向

需要先拍板产品语义，再实现：

1. **标准化到 `AGENTS.md`**：Iris init/repair 在发现根目录只有 `CLAUDE.md` 而缺少 `AGENTS.md` 时，提示用户创建 `AGENTS.md` 并追加附录 A 引导段。本仓库也应补一个根 `AGENTS.md`，让 Codex 等 agent 读到同一入口。
2. **保留 `CLAUDE.md` 作为兼容文件**：不要自动改写或删除用户已有 `CLAUDE.md`；可在 UI 里明确显示“检测到 Claude 专用入口，但缺少标准 `AGENTS.md`”。
3. **测试补齐**：给 `project.init` 增加用例：已有 `CLAUDE.md` + 无 `AGENTS.md` 时，确认 `AGENTS.md` 被创建，`CLAUDE.md` 未被改写。
4. **可选增强**：init 结果和界面文案从“AGENTS.md”泛化为“标准 agent 入口”，但底层仍以 `AGENTS.md` 为协议推荐入口，避免 Iris 变成持续维护多家 agent 私有文件的工具。

## 验收标准

- [ ] 在只有 `CLAUDE.md`、没有 `AGENTS.md` 的项目里运行 Iris init/repair 后，根目录出现带 Iris 引导段的 `AGENTS.md`。
- [ ] 既有 `CLAUDE.md` 内容不被改写。
- [ ] 重复运行 init/repair 不重复追加引导段。
- [ ] 本仓库补齐根 `AGENTS.md` 后，用 Codex 打开任意 `.iris/issue/*.md`，首轮不手动粘贴上下文也能知道应该读 `.iris/CONVENTIONS.md` 和当前 `FOCUS_DOC`。
