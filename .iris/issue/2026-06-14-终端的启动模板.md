## 调研结论（Iris「Agent 模板」vs Marina「Template」）

逐文件比对了两边的数据模型、设置 UI、会话生成链与持久化。Iris 把模板叫 `settings.agents`（住在 `~/.iris/settings.json`），Marina 叫 `Template`（住在独立的 `templates.json`）。定义书第 124 行明确 `templates` 是「给 App 读的软件配置」的合法公民，所以做一套模板系统不与协议冲突。

### 你报告的两个症结 —— 根因已定位

**1. 数量少（开箱只有 claude + 终端）**

* 现状：默认 `settings.agents` 只播种了两条 —— `claude`（hook 注入）与 `终端`（纯 shell）。见 `src/main/settings-manager.ts` DEFAULT\_SETTINGS。

* 但代码里 `AGENT_PRESETS` 其实备了 8 条（claude / codex / gemini / qwen / cursor / aider / goose / 终端，见 `SettingsView.tsx`），只是它们没进默认，要去「设置 → Agents → 快速添加」手动点进来。所以**开箱体验只有 2 个**。Marina 开箱内建 4 个（shell / claude-code / codex / opencode）。

* **隐藏的真坑（关键发现）**：就算把默认改多，老用户也吃不到。`settings-manager` 的 load 是 deep-merge，但**数组是整体替换不是逐项合并**——老用户的 `settings.json` 里已存在 `agents` 数组，新默认根本不会下发。Marina 用 `mergeTemplates` 在每次启动补全缺失的内建项（按 id 合并、保留用户改动、自动补回被删的内建）。所以「数量少」不是改一行默认能根治的，需要一次「内建补全/播种」逻辑。

**2. 不准确（claude / 终端 这组不对味）**

* 可能含义：(a) codex / gemini 这类常用 agent 该开箱即在；(b)「终端」这个 label 语义模糊（它其实就是「裸 shell，无注入」）。两点都指向「默认集合该重新策展」。

### 数据模型逐项对标

每条格式 = 维度 / Iris 现状 / Marina / 评估。

* **id** / 有 / 有 / 一致。

* **label·name** / 有 / 有 / 一致。

* **command** / 单字符串、参数内联（走 `pwsh -Command "<command>"`）/ command + `args[]` 分离 / Iris 单串经 shell 解析已够，**不必迁** args 分离（红线：壳保持哑，命令就是用户自己 shell 里的一行）。

* **icon** / 无（菜单纯文本「用 X 打开」）/ emoji 图标 / 右键菜单纯文本可接受；但**根页/右栏的「启动面板」网格值得加图标**，识别成本低。

* **env（每模板环境变量）** / **无** / `Record<string,string>`，最高优先级覆盖 / **真缺口**：按模板注入 `ANTHROPIC_MODEL` / `ANTHROPIC_BASE_URL` / 代理等，是「同一台机上 claude 跑不同模型/中转」的刚需。涉及密钥红线，需拍板（见下）。

* **shellFirst（先起 shell 再 exec）** / 无（永远 `-Command`，命令一退 shell 跟着退）/ 有 / 与 postExitAction 联动。

* **postExitAction（退出后行为）** / **无**（agent 一退，会话进 `exited` 死终端）/ `close_session` / `keep_shell` / `hold` / **真缺口**：跑完 `claude` 想回落 shell 继续敲命令、而不是面对一个死终端。需拍板。

* **injection（hook / flag / none）** / **有，且 Marina 没有这个概念** / 无 / Iris 在「上下文注入」这一维**领先** Marina —— 这是 FOCUS\_DOC 手势的核心，保留。

* **isBuiltin + 内建保护/合并** / 无（全可删、留底 1 个）/ 有（内建不可删、升级自动补回）/ Iris 更简单，但正是上面「升级不下发」隐患的来源。

* **全局默认模板 defaultTemplateId** / 无（每次都从菜单手选）/ 有 / Iris 的核心手势就是「右键手选」，**可不迁**；除非根页想要「+ 一键开默认」。

* **每书签默认模板** / 无 / 有（bookmark.defaultTemplateId）/ Iris 是文档中心非书签模型，**不适用**。

* **导出脱敏** / 无 / 有（env 含密钥时导出弹隐私确认）/ 仅当采纳 env 才需要配套。

### 判为不适用 Iris、不迁的 Marina 特性

`args[]` 分离、shellFirst 双模式（Iris 只需「退出后行为」一个维度）、每书签默认模板、SSH 远程模板执行、多窗口 owner —— 均属 Marina 产品形态，Iris 不需要。

### 定稿方案（2026-06-14 决策已锁定）

**决策记录**

* 默认开箱集合：**claude / codex / gemini / 终端**（常见三件套 + 裸终端）。

* agent 退出后行为：**回落 shell（keep-shell）**。

* 每模板 env 注入：**暂不做**。

* 补全机制：**不做运行时合并**——只有当前这一台机一份 `~/.iris/settings.json`，写个一次性迁移脚本直接补齐即可，不进 App 运行时。

**实施清单**

1. **默认集策展** —— `settings-manager.ts` 的 `DEFAULT_SETTINGS.agents` 改为四条：claude / codex / gemini（均 hook 注入）+ 终端（none）。沿用现有 `AGENT_PRESETS` 定义。
2. **一次性迁移脚本** —— 读现有 `~/.iris/settings.json`，按 id 补进缺失的 codex / gemini（保留已有项与用户改动），原子写回。仅本机跑一次，不进运行时逻辑。
3. **退出后回落 shell** —— `AgentConfig` 加 `onExit?: 'keep-shell' | 'close'`（默认 keep-shell；纯终端本就是 shell，此字段无所谓）。`resolveHostShell` 的 `buildArgs` 改造：keep-shell 时 pwsh 用 `-NoExit -Command "<cmd>"`、cmd 用 `/k`、bash 用 `-c '<cmd>; exec "$SHELL" -l'`，命令退出后落回交互 shell。设置 UI 的 Agents 表加一列「退出后」（Segmented：回落 / 关闭）。
4. **env 注入** —— 本轮跳过。
5. P3 启动面板图标 —— 本轮不在范围。

完成后 ✋ 手工验收：右键任意文档应看到 claude / codex / gemini / 终端 四项；用 claude 打开、在里面退出 claude，应落回 pwsh 提示符而非死终端。

### 实施结果（已完成）

* 默认集：`settings-manager.ts` `DEFAULT_SETTINGS.agents` 扩为 claude / codex / gemini（hook + keep-shell）+ 终端（none）。

* 退出后回落 shell：`types.ts` `AgentConfig` 加 `onExit?: 'keep-shell' | 'close'`（默认 keep-shell）；`session-manager.ts` `resolveHostShell.buildArgs(command, keepShell)` 改造——pwsh `-NoExit`、cmd `/k`、bash `; exec "$SHELL" -il`；`createSession` 按 `agent.onExit !== 'close'` 透传。`settings-manager.ts` 校验补 onExit 合法值。

* 设置 UI：`SettingsView.tsx` Agents 表加「退出后」列（Segmented：回落 / 关闭；纯终端显示「—」）。

* 迁移：`scripts/migrate-agents.mjs` 一次性按 id 给现有 `~/.iris/settings.json` 补 codex / gemini，已对本机 `~/.iris` 与 `~/.iris-dev` 各跑一次成功。

* env 注入、启动面板图标：本轮按决策跳过。

* 验证：`npm run typecheck` 干净；`npm test` 47 passed。
