# 探讨：上下文注入和联动机制

按照之前的方案，其实就是注入一个环境变量，然后让AI自己阅读，但是我认为这样其实不太好，AI会去阅读很多轮，浪费时间，浪费token，有没有办法可以在起终端的时候，将上下文直接注入

---

## 调研与方案（2026-06-12，Claude 调研后追加）

### 先修正一个前提

此前讨论隐含假设"任何启动注入都会让 agent 立刻开跑一轮"。查证后确认这是错的：主流 agent CLI 普遍有**零轮次注入通道**——内容在首条用户消息之前进入上下文，agent 依然静止等待指令。"打开 ≠ 开跑"保得住，注入与裸启动并不天然冲突。

另一点账要算清：文档内容的 token 无论谁读都要进上下文窗口，注入不省钱；它省的是**往返轮次的延迟**和**迷路探索的浪费**，并把加载时机从"用户敲第一个任务之后"提前到"开终端那一刻"。

### 关键发现：SessionStart hook 已成跨厂商事实标准

Claude Code 首创的形态（会话启动时执行用户脚本，stdout 或 JSON 的 `additionalContext` 字段直接进入上下文，不消耗模型轮次）已被各家跟进，连字段名都基本一致：

| 工具 | 零轮动态注入通道 | 备注 |
| --- | --- | --- |
| Claude Code | `SessionStart` hook；`--append-system-prompt[-file]` | hook 配 `compact` matcher 可在上下文压缩后重注入 |
| Codex CLI | `SessionStart` hook（2026 已 GA）；`-c developer_instructions=...` | 项目级 hook 首次需 `/hooks` 信任审核 |
| Gemini CLI | `SessionStart` hook `additionalContext` | 注入为历史首轮 |
| Qwen Code | `SessionStart` hook（Claude 同款命名与字段） | 默认就自动读 AGENTS.md |
| Cursor CLI | `sessionStart` hook `additional_context` | 兼容 Claude Code hooks |
| Goose | 无 hook；`goose run --system "..."` 追加系统指令 | |
| aider | 无 hook；`--read <文件>` 启动时挂文件进上下文 | 唯一连 AGENTS.md 都不自动读的 |
| opencode | 无启动注入 hook；`opencode.json` 的 `instructions[]` 支持 `{env:VAR}`/`{file:path}` 替换 | 准动态 |
| Crush | 无（仅 PreToolUse 逐工具调用注入） | 自动加载的上下文文件最广 |
| Copilot CLI | hooks 存在但输出被忽略，无法注入 | |
| Amp | 未确认 | |

普适性结论：**hook 通道覆盖头部五家**（Claude / Codex / Gemini / Qwen / Cursor）；没 hook 的 goose、aider 恰好都有等价 CLI flag；完全没有零轮动态通道的只剩 Crush、Copilot CLI、Amp。另外除 aider 外所有工具都自动加载 AGENTS.md（或可配置成它）——定义书"AGENTS.md 引导段"那一跳的地基比当初设想的更稳。

还有一个结构性事实：自动加载的上下文文件（AGENTS.md 等）是**每项目**一份、静态的；而焦点元数据是**每进程**的（同一项目可同时开多个终端、各聚焦不同文档）。所以零轮注入必须走每进程通道——环境变量、argv flag、或"hook 读环境变量"。这正好印证了当初选环境变量做协议底座的判断。

### 方案：协议不动，注入做成适配器

- **协议层（不变）**：壳只设环境变量 `FOCUS_DOC`。它仍是唯一 agent 无关、随进程生灭、可降级的真相通道（没被读，人补一句话就行）。
- **渲染层（新增，机器级）**：`~/.iris/` 放一个 `focus-context` 脚本，读 `FOCUS_DOC` 与 cwd，输出元数据块，例如：

  ```text
  [Iris] 本终端由 Iris 启动
  项目根: E:\projects\iris
  聚焦文档: .iris/misc/探讨：上下文注入和联动机制.md（类型 misc，工作区 .iris/）
  项目宪法: .iris/CONVENTIONS.md
  —— 以下为聚焦文档启动时快照，写回以盘上文件为准 ——
  <文档全文>
  ```

- **适配层（每 agent 一行配置）**：有 hook 的五家，hook 都调这同一个脚本；没 hook 的用启动模板 flag（goose `--system`、aider `--read $FOCUS_DOC`）。认识"哪个 agent 吃哪种参数"的知识住在用户的机器层模板里，壳保持机制哑。

### 两个选项（hook 与启动 flag）都用吗？

同一个 agent 上**只走一条通道**，否则同样内容注入两份。取舍次序：

1. 有 hook 的 agent 走 hook：一次安装全项目生效、壳零改动、且 compact matcher 顺带解决长会话压缩后焦点丢失——这是 flag 做不到的。
2. 没 hook 的 agent 走启动模板 flag。
3. 两者皆无的 agent 降级回现状：按 AGENTS.md 引导自己读，协议本来就允许。

hook 配置属于用户自己的 agent 配置层，Iris 只在向导里建议并代写（经用户确认），与追加 AGENTS.md 引导段同一姿势。

### 待拍板的产品决策

1. 定义书 §核心手势"裸启动（不传任何 prompt）"需修订措辞：区分"不传任务 prompt"（坚持）与"可经用户配置注入上下文"（放行）。
2. 注入内容是"元数据 + 指针"还是连同文档全文快照？建议默认带全文（省掉首轮读取，token 总量不变），超过尺寸上限（如 32 KiB，Codex 对 AGENTS.md 的同款默认）退化为只注指针。
3. hook 的分发形态：向导检测到对应 agent 后代写配置，还是只出文档教用户自装。

### 附：SessionStart hook 是什么

**Hook（钩子）**：主流 agent CLI 在自己的生命周期里留的挂钩点——会话启动时、用户提交消息前、工具调用前后、上下文压缩时等。用户在配置文件里声明"在某个挂钩点替我执行这条 shell 命令"，CLI 跑到那个时刻就调起它。这是 CLI 层的确定性功能，与模型无关。

**SessionStart 的特殊之处**：它在会话刚建立、模型尚未被调用的时刻触发，且约定**脚本往 stdout 打印的文本会被直接塞进对话上下文**，排在第一条用户消息之前。完整链路：

1. 开一个 `claude` 终端；
2. CLI 启动时发现配置里有 SessionStart hook，执行脚本；
3. 脚本打印的内容（焦点文档元数据 + 正文）被放进上下文；
4. CLI 安静停在输入框等用户——模型一次都没被调用，零轮次；
5. 用户敲下第一个任务时，模型看到的输入 = 注入的上下文 + 任务。

"零轮次"的原因：内容进上下文发生在 CLI 层，不需要模型先花一轮读文件，也不像传启动 prompt 那样逼模型立刻回复。

**配置示例**（Claude Code，`~/.claude/settings.json`，机器级全项目生效）：

```json
{
  "hooks": {
    "SessionStart": [{
      "hooks": [{
        "type": "command",
        "command": "if ($env:FOCUS_DOC) { Get-Content $env:FOCUS_DOC }"
      }]
    }]
  }
}
```

hook 脚本是普通进程，**能读到 Iris 起终端时注入的环境变量**——这是它与 Iris 协议天然咬合的地方：壳依旧只设 `FOCUS_DOC`（协议不变），"把指针变成内容"的工作由用户装在自己 agent 配置里的 hook 完成。

**相对启动 flag 的优势**：

- 装一次处处生效：之后无论从 Iris 起终端还是手动开 agent，只要 `FOCUS_DOC` 在，注入就发生，壳零改动。
- 跨厂商通用：Claude Code / Codex / Gemini CLI / Qwen Code / Cursor CLI 五家配置写法大同小异，一个 `focus-context` 脚本可被五家复用。
- compact 重注入：Claude Code 的 SessionStart 可配 `compact` 触发源，长会话上下文压缩后 hook 再执行一次，重新注入焦点文档，解决"聊久了忘焦点"——启动 flag 只在开场生效一次，做不到。

**代价**：hook 配置住在用户的 agent 配置文件里（如 `~/.claude/settings.json`），不归 Iris 管。Iris 能做的是向导检测到对应 agent 后建议并**经确认代写**——与"经确认向 AGENTS.md 追加引导段"同一姿势，不破坏"壳保持哑"的红线。
